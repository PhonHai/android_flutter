# 精简构建纪律

> **来源**：借鉴 GitHub Ponytail 项目（"the laziest senior dev"）的懒人阶梯，适配 Android/车机。
>
> **核心理念**：最好的代码是从没写过的代码。懒 = 高效，不是草率。你在 3 点被叫醒过修过度设计的代码，所以默认拒绝"为以后"的脚手架。

---

## 0. 先读懂，再精简（不可跳过的守卫）

> 懒缩短的是**方案**，不是**理解**。跳过理解去交一个小 diff，是把 bug 包装成效率。

写代码前必须：
1. 读用户要改的任务 + 它触及的每个文件
2. trace 真实调用流 end-to-end（用 code-locate 查 `.agent/project-map.md` 索引）
3. 确认理解了"这次改动必须碰哪些代码"
4. **然后**才开始爬懒人阶梯

**反例（危险懒）**：用户报"蓝牙连不上"，你看了报错行就加个 try-catch 吞掉异常，3 行搞定。这是把症状埋了，根因还在。正确做法见下方"bug 修复=治根因"。

---

## 1. 懒人阶梯（写到第一级能站住就停）

按顺序自问，**停在第 1 个能解决问题的 rung**：

| Rung | 自问 | Android 场景举例 |
|------|------|-----------------|
| **1. 这功能需要存在吗？** | 投机性需求 → 跳过，一行话说明 | "要不要加个配置开关？"→ 不加，等真有第二个场景再说（YAGNI） |
| **2. 本仓库已经有了？** | helper/util/type/pattern 在别处 → **复用，别重写** | 要写日期格式化？先 grep `DateFormat` / `DateUtils`，本仓库 90% 已有 |
| **3. Kotlin/Android stdlib 有？** | 用标准库 | 要判空链？`?.let{}` / `?:`，别造 `Optional` 包装；要协程？`kotlinx.coroutines` 已在 |
| **4. 平台原生功能覆盖？** | 用平台能力，别引依赖 | 要日期选择？系统 `DatePickerDialog`，别引 flatpickr 类的库；要权限请求？系统 `ActivityResultContracts`，别造轮子 |
| **5. 已装的依赖能解？** | 用已装的，别为新功能加新依赖 | 已有 Gson？别为单字段再加 Moshi；已有 Room？别为一张表引 SQLDelight |
| **6. 能写一行吗？** | 一行就一行 | `val v = list.find { it.id == x } ?: return` 比"先声明 var 再 for 循环赋值"好 |
| **7. 都不行** | 才写"能跑的最少代码" | 此时才动键盘 |

**两条 rung 都成立 → 取更高的那条**。第一个能跑通的懒方案就是对的——前提是你已经懂了问题。

---

## 2. bug 修复 = 治根因，不治症状

> 报告说的是症状，不是根因。改前必须 grep 每一个调用方。

**懒修法 = 根因修法**：在共享函数里加一处守卫，比在每个调用方各加一处守卫的 diff 更小——而且只修报告点名的那条路径，其他兄弟调用方还在坏。

### 操作流程
1. 用 code-locate 找到被报错的函数
2. `grep -rn "函数名"` 找**所有**调用方（车机项目特别注意 AIDL 跨进程调用方）
3. 判断根因在哪一层（共享函数 vs 某个调用方传错参）
4. 根因在共享函数 → 改一处，所有调用方受益
5. 根因在某个调用方传错参 → 改那个调用方，别动共享函数（别为修一个 bug 引入兼容层）

### 对照
| 做法 | diff 大小 | 是否根治 | 评价 |
|------|----------|---------|------|
| 只修报告点名的路径 | 小 | ❌ 兄弟调用方还坏 | 反例 |
| 每个调用方各加守卫 | 大 | ✅ | 不懒 |
| 共享函数改一处 | 最小 | ✅ | ✓ 懒且根治 |

## 3. 不可违反的规则

| 规则 | 说明 |
|------|------|
| **不要未要求的抽象** | 单实现的 interface、单产品的 factory、永不变的值的 config——都不要 |
| **不要脚手架"以后用"** | 以后要用，以后自己搭。现在搭的是债务 |
| **删除优先于新增** | 能删的代码比能写的代码值钱 |
| **无聊优先于聪明** | 聪明是别人 3 点要解码的东西 |
| **最少文件** | 一个函数能搞定的别拆成 3 个文件 |
| **最短能跑 diff 胜出** | 但前提是懂了问题——最小但改错地方不是懒，是第二个 bug |
| **两个 stdlib 同样大小 → 选边界正确的** | 懒 = 写更少代码，不是选更脆弱的算法 |

## 4. 标记"故意的精简"

故意简化时，留一行 `lazy:` 注释说明天花板和升级路径（不是给 AI 看的，是给以后维护的人看的）：

```kotlin
// lazy: 全局锁，吞吐上来后改 per-account 锁
private val lock = Any()

// lazy: O(n²) 扫描，列表超 1000 项换二分
fun find(ids: List<String>): Item? = items.firstOrNull { it.id in ids }
```

**注意**：这跟 no-ai-smell 不冲突——`lazy:` 注释说明的是"技术取舍"，不是"AI 思考痕迹"。

## 5. 输出纪律

```
[代码]
→ 跳过: [X]，加的时候: [Y]
```

- 代码在前
- 之后最多 3 行短说明：跳过了什么、什么时候该加
- **不写作文，不做功能导览，不做设计笔记**
- 如果解释比代码还长 → 删解释。每段为简化辩护的文字，都是把复杂度用散文塞回来
- 用户明确要的报告/走查/分阶段说明不算债，照给——这条只针对"未要求的散文"

## 6. 不可精简的地方（即使 ultra 档也不动）

| 不可精简 | 原因 |
|---------|------|
| **信任边界的输入校验** | 车机 AIDL/Intent 入参、系统 Service 入参——精简掉就是安全漏洞 |
| **防数据丢失的异常处理** | 写文件/数据库的关键路径——精简掉就是数据损坏 |
| **安全措施** | 权限校验、签名校验、sharedUserId——精简掉是红线 |
| **无障碍基础** | TalkBack 标签、焦点顺序——精简掉是合规问题 |
| **用户明确要求的功能** | 用户要完整的就给完整的，别再争论 |

## 7. 自检（非平凡逻辑必须留一个可跑的检查）

非平凡逻辑（一个分支 / 一个循环 / 一个解析器 / 涉及钱或安全的路径）必须留下**一个**能跑的检查，最小到逻辑坏了就失败的程度：

- 一个 `assert` 风格的 `demo()` / `main()` 自检，或
- 一个小的 `test_*` 函数

不要框架，不要 fixture，不要每个函数一套——除非用户要。**一行能搞定的不需要测试**（YAGNI 同样适用于测试）。

车机项目特别：自检放 `src/test/` 或 `@JvmStatic main`，别污染 release 包。

## 8. 强度三档

| 档位 | 行为 | 何时用 |
|------|------|--------|
| **lite** | 写用户要的，但用一行指出更懒的替代方案，让用户选 | 用户不确定要不要极简时 |
| **full**（默认） | 阶梯强制执行。stdlib/原生优先。最短 diff + 最短解释 | 默认开发 |
| **ultra** | YAGNI 极端主义。删除优先于新增。交一行代码的同时质疑剩下的需求 | 用户明确要"砍到极限" |

### 示例：用户说"给这些 API 响应加个缓存"
- **lite**：「加好了。FYI：`kotlinx.coroutines` 的缓存或 Room 的 `@Query` 也能一行覆盖，不想自己维护缓存类的话说一声。」
- **full**：「在 fetch 函数加 `Mutex + HashMap` 缓存。跳过自定义缓存类，等 `HashMap` 测出瓶颈再加 LRU。」
- **ultra**：「没 profiler 数据前不加缓存。真要加时用 Room `@Query` 缓存。手撸 TTL 缓存类是个 hit rate 还行的 bug 农场。」

切档：用户说"懒人 lite" / "懒人 full" / "懒人 ultra"。
关闭：用户说"停止懒人" / "正常模式"。

## 9. 与其他 skill 的关系

| Skill | 关系 |
|-------|------|
| `code-locate` | 本 skill 的 Rung 2（复用现有）依赖 code-locate 找到已有代码 |
| `confirm-first` | confirm-first 输出方案 → 用户同意 → 进入本 skill 决定怎么写 |
| `coding-rules` | 本 skill 决定"写多少"，coding-rules 决定"写成什么样"（风格/最小改动/去AI味） |
| `verify-loop` | 本 skill 写完 → verify-loop 验证。本 skill 的"留一个自检"会被 verify-loop 跑 |
| `git-commit` | verify-loop 通过 → git-commit 提交 |

**完整链路**：
```
project-memory(懂项目) → code-locate(找代码) → arch-rules(架构约束) 
→ confirm-first(提方案) → lazy-build(决定写多少) → coding-rules(决定怎么写) 
→ verify-loop(验证) → git-commit(提交)
```

## 10. Android/车机特有提醒

| 场景 | 懒人做法 |
|------|---------|
| 要 IPC？先看本仓库 AIDL 有没有现成的 | 别为一次性调用新增 AIDL 接口 |
| 要系统权限？先看 `AndroidManifest.xml` 已申请的 | 别重复申请，别加用不到的 |
| 要 Car API？先 grep `android.car.*` 看本仓库封装 | 别直接调 `CarPropertyManager`，找本仓库的 wrapper |
| 要 Launcher3/Settings 的某功能？先看本仓库有没有继承/复用 | 别从头实现 |
| 要 Resources？用 `res/values/` 已有的 string/color/drawable | 别硬编码，别新建重复资源 |

车机物理世界提醒（来自 Ponytail 的硬件边界）：真实传感器有漂移、真实时钟不准、PCA9685 会快几个百分点——**保留校准旋钮，别为了精简删掉调参入口**，物理世界需要的微调是极简模型看不见的。

## 11. 用户怎么用（使用时机——agent 必读，避免用户困惑）

> 用户最常问的问题："懒人模式什么时候说？是在 AI 给方案时说，还是同意方案后说？"

### 三种时机（按推荐度排序）

**时机 1（推荐）：提需求时一起说** — 影响方案，最省事

```
用户：帮我加个蓝牙连接状态监听，懒人ultra
```
AI 在方案阶段就走懒人阶梯，输出的方案本身就是精简的，省一轮往返。

**时机 2（次选）：看完方案同意时补一句** — 救场用

```
用户：帮我加个蓝牙连接状态监听
AI：[输出 80 行 ApiCacheManager 单例方案]
用户：同意，但用懒人模式写
AI：[重新走阶梯，发现本仓库已有封装，改成 5 行复用方案]
```

**时机 3（默认）：不说** — full 档自动加载

日常开发，AI 写代码时自动走阶梯。用户想切 lite/ultra 或关闭时才需要说。

### agent 行为规则

- 用户提需求时**带**懒人档位 → 方案阶段就按该档走（时机 1）
- 用户提需求时**不带** → 默认 full 档，在 confirm-first 方案里说明"已按懒人 full 走阶梯"；用户同意方案时可补档位（时机 2）
- 用户说"停止懒人" → 本会话剩余部分关闭 lazy-build，回到普通模式
- 用户说"懒人lite/full/ultra" → 切档，本会话剩余部分保持该档（除非再切）

### 给用户的提示（confirm-first 方案卡里附带）

输出方案时，方案末尾加一行（如非默认 full 档）：
```
懒人档位: full（默认）。想更精简说"懒人ultra"，想看替代方案说"懒人lite"
```
