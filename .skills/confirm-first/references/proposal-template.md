# 修改方案输出模板

> agent 改代码前必须输出此卡片。4 章节不可省略。

---

## 完整版（标准修改）

```markdown
📋 修改方案

【文件】<相对路径>
【位置】<类名.方法名, line X-Y>
【改前】
<原代码 1-5 行>
【改后】
<新代码 1-5 行>
【改动】<N 行>

🤔 修改原因
  <为什么改？技术原因>
  方案对比:
  - 方案A(推荐): <做法> — <选它的理由>
  - 方案B: <做法> — <没选的理由>

🎯 影响范围
  - <文件/模块列表>
  - <是否动 Manifest/Gradle/AIDL>
  - <是否影响其他 flavor/模块>

⚠️ 风险评估
  风险等级: <低/中/高>
  潜在问题: <可能引入什么 bug>
  验证方式:
    1. <构建命令>
    2. <测试命令>
    3. <手动验证步骤>
  回滚方式: <git revert / 手动还原>

  同意我就改?
```

---

## 简化版（小改动：单文件 + <5 行 + 不动 manifest/gradle）

```markdown
📋 修改方案（小）

【文件】<路径> 【位置】<方法, line X>
【改前】<原代码>
【改后】<新代码>
【原因】<一句话>

🎯 影响: 单文件，不动 Manifest/Gradle
⚠️ 风险: 低。验证: <命令>。回滚: git revert

同意就改?
```

> 注意：简化版仍要有"影响"和"风险"两节，只是各一句话。不可完全省略。

---

## 复杂版（大改动：多文件 / 动核心 / 重构）

```markdown
📋 修改方案（复杂）

【涉及文件】
1. <路径1> — <改什么>
2. <路径2> — <改什么>
3. <路径3> — <改什么>

【实施步骤】
1. 先改 <文件1> 的 <方法>，因为 <依赖关系>
2. 再改 <文件2>，同步 <接口>
3. 最后改 <文件3>，更新 <注册/Manifest>

🤔 修改原因
  <详细背景>
  方案对比:
  - 方案A(推荐): <做法>
    优点: ...
    缺点: ...
  - 方案B: <做法>
  - 方案C: <做法>

🎯 影响范围
  - <模块级影响>
  - <是否破坏向后兼容>
  - <是否需要数据迁移>

⚠️ 风险评估
  风险等级: <高>
  潜在问题:
    1. <风险1 + 缓解>
    2. <风险2 + 缓解>
  验证方式:
    1. <单元测试>
    2. <集成测试>
    3. <手动验证>
  回滚方式: <git revert + 数据回滚步骤>
  回滚风险: <回滚本身的成本>

  建议分步提交：
  - commit 1: <文件1>
  - commit 2: <文件2>
  - commit 3: <文件3>

  同意我就按步骤改?
```

---

## 关键原则

1. **4 章节不可省略**（方案/原因/影响/风险）——简化版也要有，只是各一句
2. **改前改后都要贴代码**——让用户直观看到差异
3. **最后一句话必须是询问**："同意就改?"/"同意我就改?"——制造明确的决策点
4. **输出在对话里**，不写文件——方案是临时讨论，不是文档
5. **风险等级要诚实**——低就低，高就高，别为显得安全而压低

---

## 何时用哪版

| 改动特征 | 用哪版 |
|---------|--------|
| 单文件 + <5 行 + 不动 manifest/gradle/aidl | 简化版 |
| 单文件 + 5-30 行 / 动非核心 | 完整版 |
| 多文件 / 动核心逻辑 / 重构 / 动 manifest | 复杂版 |

> 弱模型注意：拿不准用哪版 → **用完整版**（信息多比少好）。
