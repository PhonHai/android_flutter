# 闭环验证（Build-Verify Loop）

> **来自 harness engineering 实践**：模型有"first plausible solution bias"——对第一个看起来合理的方案产生偏见，不主动验证等于开环控制。

**ROLE**：Build → 读完整输出 → 对照需求比较 → 失败则修复重试（最多 2 次）。

**WHEN**：任何 Edit/Write 代码后自动触发。用户说"跳过验证"/"不检查"/"别跑构建"时豁免。

---

## 为什么需要这个 skill

agent 改完代码后的典型失败模式：
1. 写完 → 重读自己的代码 → "看起来对" → 停止
2. **没有实际跑构建/测试**
3. 即使跑了，没**对照原始需求**比较

---

## 强制流程

### Step 1：读原始需求
明确"用户到底要什么"。

### Step 2：跑构建
按 config 的 `verify_loop.build_cmd`。**必须读完整输出**，不能只看最后一行。

### Step 3：判定结果

**✅ 构建成功** → 进入 Step 4

**❌ 构建失败** → 进入修复循环：
1. 读错误输出，定位失败原因
2. **回溯原始需求**，确认修复方向
3. 修复代码
4. 重新跑构建
5. 最多重试 2 次
6. 超过次数 → STOP，报告用户

### Step 4：对照原始需求比较

```
□ 用户要的功能/修复，代码实现了吗?
□ 实现方式符合用户的约束吗?
□ 有没有引入用户没要求的行为?
□ 边界情况处理了吗?(null/空列表/异常)
```

### Step 5：跑 harness 检查
```bash
bash .skills/_harness/scripts/diff_guard.sh
bash .skills/_harness/scripts/ai_smell_check.sh <改的文件>
```

### Step 6：更新 progress.json
记录验证结果。

### Step 7：报告用户
```
✅ 验证通过
- 构建: 成功
- 需求对照: 实现了 <用户要的>，未引入 <用户没要求的>
- harness: diff_guard / ai_smell 均通过
```

---

## 循环检测（防 doom loop）

若同一文件被 Edit 超过 3 次仍在修构建错误：
```
⚠️ 已对 <文件> 编辑 N 次仍未通过验证。
这可能是方案问题，不是实现问题。
建议：停下来，重新审视方案，或问用户。
```

---

## 推理"三明治"策略

- **规划阶段**（confirm-first）：高推理，充分理解问题
- **实现阶段**（Edit）：中等推理，节省 token
- **验证阶段**（本 skill）：高推理，捕获错误

---

## 何时可跳过

- 用户明确说"不用验证"
- 纯文档修改（非代码）
- config 里 `verify_loop.enabled: false`

**其他情况不可跳过。** agent 没有判断"这个改动太简单不需要验证"的权限。
