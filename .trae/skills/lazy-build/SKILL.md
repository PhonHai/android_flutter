# 精简构建纪律

> **核心理念**：最好的代码是从没写过的代码。懒 = 高效，不是草率。

**ROLE**：决定"写多少代码"——7 级阶梯从 YAGNI 到最小 diff，bug 治根因，停在第 1 个能解决问题的 rung。

**WHEN**：写新代码前默认 full 档；用户说"懒人模式/lite/ultra"切档，说"停止懒人"关闭。

**INTENSITY**：lite=写+提供更懒替代方案 / full(默认)=阶梯强制执行 / ultra=YAGNI 极端，删除优先于新增。

---

## 0. 先读懂，再精简（不可跳过）

写代码前必须：
1. 读用户要改的任务 + 它触及的每个文件
2. trace 真实调用流 end-to-end
3. 确认理解了"这次改动必须碰哪些代码"
4. **然后**才开始爬懒人阶梯

---

## 1. 懒人阶梯（写到第一级能站住就停）

| Rung | 自问 | Android 场景举例 |
|------|------|-----------------|
| **1. 这功能需要存在吗？** | 投机性需求 → 跳过 | "要不要加个配置开关？"→ 不加，等真有第二个场景再说 |
| **2. 本仓库已经有了？** | 复用，别重写 | 要写日期格式化？先 grep，本仓库 90% 已有 |
| **3. Kotlin/Android stdlib 有？** | 用标准库 | 判空用 `?.let{}` / `?:`，别造 `Optional` 包装 |
| **4. 平台原生功能覆盖？** | 用平台能力 | 日期选择用系统 `DatePickerDialog`，别引库 |
| **5. 已装的依赖能解？** | 用已装的 | 已有 Gson？别为单字段再加 Moshi |
| **6. 能写一行吗？** | 一行就一行 | `val v = list.find { it.id == x } ?: return` |
| **7. 都不行** | 才写"能跑的最少代码" | 此时才动键盘 |

---

## 2. bug 修复 = 治根因，不治症状

1. 找到被报错的函数
2. `grep -rn "函数名"` 找**所有**调用方
3. 判断根因在哪一层
4. 根因在共享函数 → 改一处，所有调用方受益
5. 根因在某个调用方传错参 → 改那个调用方

---

## 3. 不可违反的规则

| 规则 | 说明 |
|------|------|
| **不要未要求的抽象** | 单实现的 interface、单产品的 factory 都不要 |
| **不要脚手架"以后用"** | 以后要用，以后自己搭 |
| **删除优先于新增** | 能删的代码比能写的代码值钱 |
| **无聊优先于聪明** | 聪明是别人 3 点要解码的东西 |
| **最少文件** | 一个函数能搞定的别拆成 3 个文件 |
| **最短能跑 diff 胜出** | 但前提是懂了问题 |

---

## 4. 标记"故意的精简"

故意简化时，留一行 `lazy:` 注释：
```kotlin
// lazy: 全局锁，吞吐上来后改 per-account 锁
private val lock = Any()
```

---

## 5. 不可精简的地方（即使 ultra 档也不动）

| 不可精简 | 原因 |
|---------|------|
| **信任边界的输入校验** | 精简掉就是安全漏洞 |
| **防数据丢失的异常处理** | 写文件/数据库的关键路径 |
| **安全措施** | 权限校验、签名校验 |
| **用户明确要求的功能** | 用户要完整的就给完整的 |

---

## 6. 强度三档

| 档位 | 行为 |
|------|------|
| **lite** | 写用户要的，但用一行指出更懒的替代方案 |
| **full**（默认） | 阶梯强制执行，最短 diff + 最短解释 |
| **ultra** | YAGNI 极端主义，删除优先于新增 |

切档：用户说"懒人 lite" / "懒人 full" / "懒人 ultra"。关闭：用户说"停止懒人"。
